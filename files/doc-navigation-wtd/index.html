New company ... same problem

How do you increase the user's ability to find answers in docs?

Design principles to tackle the problem

  • Hierarchy

  • Design principles inform decisions

Action 1: Group topics in product sidebars

  • All content on one web CMS
  • No way to know entirety of docs for one product
  • Lists of inline links at bottom of topics

Design principle: Hierarchy

"Hierarchical organization is the simplest structure for visualizing and understanding complexity."
— Universal Principles of Design

  • Hierarchy usually presented in sidebar nav
  • Common in tech docs, but hard to get right
  • A tool for simplifying complexity

Hierarchy example: Google

  • Right size - not overwhelming number of topics
  • Can see them all at a glance
  • Appears on left side in predictable space

Hierarchies break down complexity

  • A tool for reducing complexity of a system
  • Functions like a list of parts
  • We break complex systems down into parts (theorum, company, API, animals, conference)

Hierarchies give idea of "whole"

  • Grounds users spatially and mentally
  • Experience of getting rid of it -- users hate it.
  • Makes sense when product has sense of whole, not KB

Deep hierarchies don't relay "whole"

  • Deep nesting prevents users from seeing whole
  • Expand/collapse best if expand all / collapse all buttons
  • Figure out the right scope for the material

Hierarchies show what's in the neighborhood

  • Unknown unknown
  • Lets users know what's there - like having a map of city
  • Common usage of sidebar is for topics related to present topic

Action 2: Navigate from doc set to doc set

  • When you reduce product scope in sidebar, need mechanisms to nav among doc sets
  • Related links section below nav
  • Related links is curated scope, not comprehensive

Design principle: Progressive disclosure

"A strategy for managing information complexity in which only necessary or requested information is displayed at any given time."

— Universal Principles of Design
  • Don't show all content to user, but layer at diff. levels
  • Often used in CSH
  • Can place more topics at page level, not necessary to list in sidebar

Layer information at different levels

Doc home example: Azure

  • Card pattern is common
  • Think about what users need to know at different points in their journey
  • Need easy way to return to this index page from any doc set

Doc home example: AWS

  • Enables quick scanning above the fold
  • Pathfinding page, train station terminal routing
  • Easier to implement -- no need for graphics

Prod. home example: Harvest

  • Implemented without sidebar hierarchy
  • Present most popular, common topics first
  • Works well for larger KB type systems

Action 3: Allow navigation within content

  • Users can be blind to sidebar
  • Inline links provide more context/meaning to links
  • Don't be shy about inline links

Design principle: Immersion

"A state of mental focus so intense that awareness of the 'real' world is lost, generally resulting a feeling of joy and satisfaction."
— Universal Principles of Design

  • Think theme parks, movie theaters, stores
  • Might be a stretch to apply to help docs, but ...
  • Primary application --> bring help to user within system

Even in help, users want to stay inline

  • Think of sidebar as "outside" of content
  • User's focus is inline
  • No need to switch contexts

Bottom-up navigation

  • If you tell me I can do something, link to how to do that something.
  • If you tell me I can use something, link to a description of that something.
  • If you mention a concept or an idea, link to a description of that concept or idea.
— Mark Baker
  • Practice application of EPPO
  • Sure we do this already, but not like Wikipedia
  • Challenges: decision points in content, valid links, authoring with so many links

Immersion example: Wikipedia

  • Automated or managed way of linking (e.g, indirect references) is best
  • Jumping off points not a concern -- if user is unsatisfied, may be in wrong place to begin with.
  • Scent of information -- inline links point to highly relevant, related content

Action 4: Make popular topics easy to access

  • Decisions based on metrics
  • 80/20 rule. 80% of traffic is on 20% of pages. Make easier for users to find/view those pages.
  • Place in multiple places

Design principle: Desire Line

"Traces of use or wear that indicate preferred methods of interaction with an object or environment." — Universal Principles of Design

  • Formalize natural paths/desires
  • Central Park pathways
  • This principle connects to a larger philosophy

Desire line example: Twitter

  • What are my top 6-10 most common topics?
  • Most popular pages listings
  • Get this information by analyzing metrics

Action 5: Reduce information fragmentation

  • My pet peeve is tiny topics
  • Content didn't really have this problem
  • Constant questoin -- can I consolidate, reduce # of pages?

Design principle: Modularity

"A method of managing system complexity that involves dividing large systems into multiple, smaller self-contained systems."
— Universal Principles of Design

  • Provides efficiencies for authoring
  • Not all content can be divided like this

Users read non-sequentially

You can just read the sections about the tasks you want to do.
— John Carroll, The Nurnberg Funnel
  • Reacting against sequential learning modules
  • Help must support entry at any point

Create self-contained "articles"

  • Don't be afraid of longer topics.
  • break the topic when it's too confusing to have all the info on the same page
  • people can navigate a single page if you have toc

Building blocks â‰  presentation

  • task, concept, reference --> doesn't have to be mini-topics
  • don't force users to be pinballs

Action 6: Hang signposts along the way

  • like hiker on a path
  • context before
  • big believe in next steps

Design principle: Wayfinding

"The process of using spatial and environmental information to navigate to a destination."
— Universal Principles of Design

  • users have big picture, now they need signposts along the way
  • links at start (for context) and end (next steps)
  • give users a map to follow for their tasks

Provide context before

  • this only works for simple, linear processes
  • you could also consolidate on same page with expandable sections
  • more realistic is to have a complex workflow map with many decision trees

Provide context after

  • not all tasks have next steps, but suggest one anyway
  • most tasks are situated inside a larger process
  • if you can gamify the process with a progress bar, even better!

Breadcrumbs (Facebook)

  • welcome addition for context
  • not always easy to implement
  • avoid constructing your entire nav through breadcrumbs
